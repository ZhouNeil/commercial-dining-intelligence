# 餐厅推荐系统：结构、技术与数据说明

本文档面向当前代码实现，概括**餐厅检索与推荐（MVP）**的模块结构、所用技术、核心方法与数据/产物路径，便于维护与答辩说明。

---

## 1. 总体定位

系统在 Streamlit 中提供「自然语言 + 高级筛选」的餐厅推荐：将用户输入解析为**州/城市/预算/地标与半径/菜系**等约束，在离线构建的 **TF-IDF 文本索引**上做 **余弦相似度检索**，再经**多因子加权重排**得到 Top-K，并附带基于规则的**简短洞察**（非 LLM）。可选接入 **Google Maps 清洗表**与 Yelp 数据一并建索引。

> 说明：`docs/d1doc.md` 中部分表述（如「dense embedding 为主检索」）偏设计愿景；**当前主线实现以稀疏 TF-IDF + 余弦为主检索**。`sentence-transformers` 在 UI 中仅用于**在规则未解析出州名时，可选地推断美国州**（`all-MiniLM-L6-v2`），不参与餐厅向量检索。

---

## 2. 推荐模型结构（总览）

当前实现**不是**独立的「训练好的深度推荐模型」，而是一条固定的**稀疏语义检索 + 约束过滤 + 多信号加权排序**流水线，结构可概括为：

| 层级 | 作用 | 实现要点 |
|------|------|----------|
| **查询侧** | 把自然语言与 UI 选项变成「检索文本 + 硬约束」 | `parse_query` 产出 `ParsedQuery`（预算、地标经纬度、半径、`semantic_query` 等）；可选 `MiniLM` 仅补 **州** |
| **物品表示** | 每家店一个可检索向量 | 评论档案 + 元数据 → `build_profile_text` → **一个词文档** → **`TfidfVectorizer` 拟合/变换** → 稀疏向量（与 `RestaurantSearchIndex` 中行对齐） |
| **查询表示** | 与物品同一向量空间 | 拼接后的 `keywords` 走**同一** `vectorizer.transform` |
| **相似度模型** | 语义匹配强度 | 查询向量与各行 **余弦相似度** → `similarity`（手写点积 / 范数，非 `NearestNeighbors`） |
| **约束** | 缩小候选集 | 州/市精确匹配；菜系规则（`categories` 关键词，可放宽）；Haversine **距离 + 半径** |
| **排序模型** | Top-K 最终顺序 | 过滤后候选上各信号 **min-max（或固定映射）到 [0,1]**，再 **线性加权**：`final_score` 中星级项为 **`w_rating × stars_rank`**，`stars_rank` 由 `stars_norm` 与评论量可信度收缩得到（详见 §6.2）；其余项为 `w_semantic·sim_n`、`w_price·price_match` 等（侧栏可调） |
| **解释** | 每条结果短文 | `generate_insight`：规则模板，**无** LLM |

**离线 vs 在线**：向量化器与餐厅矩阵在**离线** `build_or_load_index` 中建好并落盘（`models/artifacts/`）；**在线**只做向量变换、相似度、过滤与加权排序，无反向传播或再训练。

**与「混合检索」设计文档的关系**：代码里**尚未**把餐厅/查询换成稠密 embedding 检索；若引入双塔或句向量，会在上述「物品/查询表示」层替换或并联 TF-IDF，排序层可沿用或再学一层模型。

## 3. 模块结构（代码组织）

| 路径 | 职责 |
|------|------|
| `app/main.py` | Streamlit 入口：侧栏权重、索引加载/强制重建、NL 与高级筛选 Tab、调用 `parse_query` 与 `recommend_keywords`、结果表格、Folium 地图、洞察展示 |
| `app/core/retrieval.py` | **核心**：`TouristRetrieval` — 文档构建、`RestaurantSearchIndex`、`build_or_load_index`、`recommend_keywords`（过滤 + 重排） |
| `app/core/profile_builder.py` | 从正/负向评论抽取高频主题（词/短语），拼成可解释的「档案文本」供 TF-IDF 使用（标准库为主，便于单测） |
| `app/core/google_maps_loader.py` | 读取 `google_maps_restaurants(cleaned).csv`，映射为与 `business_dining.csv` 接近的列；稳定 `gm_*` 业务 ID；合成 Google 行在无 Yelp 评论时的简介片段 |
| `app/search/query_parser.py` | 规则式 NL 解析：`ParsedQuery`（菜系、预算、cheap/moderate/expensive、州、地标坐标、半径 km、`semantic_query` 等） |
| `app/search/geo_constants.py` | 美国州名/缩写等常量（被解析器与语义州推断复用） |
| `app/search/semantic_filters.py` | 可选：MiniLM 编码查询与各州显示标签，余弦选州 |
| `app/search/insights.py` | 对每条结果生成 `why` / `pros` / `cons` 短文案（星级、评论数、类别、距离、预算与价格档一致性等启发式） |

**相关但非「推荐主链路」**：`app/pages/1_KMeans_聚类.py`、`app/clustering/kmeans_clustering.py` 属于同应用内的聚类分析页，与 TF-IDF 检索并列，不改变推荐索引逻辑。

**历史/实验**：根目录 `models/retrieval_lee.py` 为较早的 `CountVectorizer + categories` 原型，**非**当前 `app` 主流程。

## 4. 数据流（端到端）

```text
用户输入（NL + 可选高级筛选）
  → parse_query：结构化约束 + semantic_query
  → （可选）MiniLM：补全 state_code
  → TouristRetrieval.recommend_keywords
       · query → TF-IDF 向量
       · 与索引矩阵逐条余弦相似度
       · 州/市/菜系/距离硬过滤或软约束
       · 多因子加权 final_score → Top-K
  → generate_insight：每条结果的解释文案
  → Streamlit 表格 + Folium 地图
```

- **语言与运行时**：Python 3（见项目环境）
- **Web UI**：Streamlit；地图 **Folium** + `streamlit-folium`
- **数据处理**：pandas、numpy（`numpy<2`）、scipy（稀疏矩阵）

- **其他依赖**（与推荐弱相关或分析用）：matplotlib、seaborn、xgboost 等（见根目录 `requirements.txt`）

## 6. 核心方法摘要

### 6.1 索引文档如何构造

1. 读取 `business_dining.csv`，按 `review_count` 等策略选取最多 `max_businesses` 家店；可按**州**分组配额，避免索引过度集中在少数州导致过滤后无结果。
2. 若开启 `embed_google_maps` 且存在 Google CSV，则 `concat` 合并行。
3. 对入选 `business_id`，分块扫描 `review_dining.csv`：  
   - **正向**：`stars >= 4` 的评论文本，最多 `pos_limit` 条；  
   - **负向**：`stars <= 2`，最多 `neg_limit` 条。  
4. 对每家调用 `build_profile_text`：名称、类目、城市州、`attributes` + 正/负主题短语，合成**单一文档字符串**。
5. Google 行（`gm_*`）无 Yelp 评论时，用 `synthetic_google_profile_snippets` 补充正负片段。
6. `TfidfVectorizer`：`max_features`（如 50000）、`stop_words='english'`、`min_df=1`、`ngram_range=(1,2)`，拟合文档得到 `restaurant_matrix`（CSR），并保存 `vectorizer`、`meta.csv`、`restaurant_ids.npy` 等。

**版本与重建**：`index_config.json` 记录 `index_version`、采样参数等；缺失或不一致则自动重建。

### 6.2 检索与重排

- **语义分**：查询文本经同一 `vectorizer` 得到稀疏向量，与所有餐厅向量 **余弦相似度** → `similarity`。
- **硬过滤**：`state_norm`、`city_norm`（精确匹配）；可选 **Haversine** 距离 + `max_radius_km`；菜系通过 `categories_norm` 与预定义关键词表做 `contains` 匹配，若过滤为空则**放宽菜系**保留地理结果。
- **软信号归一化后加权**（侧栏可调权重）：  
  - `w_semantic`：相似度 min-max 归一化  
  - `w_rating`：`stars` 先映射到 [0,1]，再按 `review_count` 做 **可信度收缩**（评论极少时向中性分拉回；约 150+ 条评论接近全额星级），再参与加权  
  - `w_price`：从 `attributes` 解析 `RestaurantsPriceRange2`，与 budget 目标档位的匹配度  
  - `w_distance`：有参考点时，距离越近分越高（与默认/半径尺度相关）  
  - `w_popularity`：`log1p(review_count)` 再 min-max  
- 综合为 **final_score**，取 Top-K，返回列包含 `name, address, city, state, categories, stars, review_count, similarity, final_score, latitude, longitude, distance_km, price_tier, price_match` 等。

### 6.3 查询理解

- **规则 + 正则**：预算词、地标表（如 NYU、费城）、半径、`semantic_query` 剥离预算/位置噪声等。
- **可选 MiniLM**：仅在「未从文本解析出州」且用户勾选时，在**数据集中出现的州列表**上做查询–州标签相似度，带阈值与 margin，减少误触发。

### 6.4 洞察生成

- `generate_insight`：基于星级、评论数、类别截断、与查询的相似度、距离、预算与 `price_tier` 一致性等，拼出 **优点 / 缺点 / 一句话推荐理由**，无大模型调用。

## 7. 数据与产物

### 7.1 输入数据（默认相对项目根）

| 文件 | 作用 |
|------|------|
| `data/cleaned/business_dining.csv` | Yelp 餐饮商户主表（必用）：`business_id`、`review_count`、`categories`、`stars`、地理与 `attributes` 等 |
| `data/cleaned/review_dining.csv` | Yelp 评论（必用）：按店聚合评论文本；体量大，建索引时分块读取 |
| `data/cleaned/google_maps_restaurants(cleaned).csv` | **可选**：Google Maps 导出清洗表；存在则并入索引（列如 `Lat`/`Lon`/`Name`/`Address`/`Rating`/`Price Category` 等，见 `google_maps_loader`） |

**验证说明**：`business_dining` 全表约有 **上千** 个不同的 `(city, state)` 组合；当前索引默认只收录 **按商户数排序的前 11 个都会核心区**（见 `app/core/index_cities.py`，约 3.2 万+ 店，含 St. Louis 等写法变体）。更改后需 **Force Rebuild Index**。

州下拉等：`union_state_options` 会联合 Yelp 与 Google 表中的州信息。

### 7.2 离线产物（索引）

目录默认：`models/artifacts/`（与 `TouristRetrieval(..., index_dir=...)` 一致）

| 文件 | 含义 |
|------|------|
| `vectorizer.joblib` | 拟合后的 `TfidfVectorizer` |
| `restaurant_matrix.npz` | 餐厅 TF-IDF 稀疏矩阵 |
| `restaurant_ids.npy` | 与矩阵行对齐的 `business_id` |
| `meta.csv` | 商户元数据行（含 `state_norm`、`city_norm`、`categories_norm` 等） |
| `index_config.json` | 建索引参数与版本，用于判断是否重建 |

## 8. 小结

当前餐厅推荐 MVP 是一条 **「可解释档案文本 + TF-IDF + 余弦检索 + 多因子重排」** 的离线流水线，数据以 **Yelp 清洗切片**为主、**Google Maps 清洗表**可选增强覆盖；UI 集成 **规则 NL 解析、可选地语义补州、Folium 可视化与规则洞察**。若需与 `d1doc.md` 中「以稠密向量为主检索」的设计对齐，需在实现层增加查询与文档的双塔/嵌入索引，本文描述的是**仓库现状**而非规划文档本身。

# Restaurant Recommendation System — Interactive Upgrade (v2)

## 1. Goal

Upgrade the current **single-pass retrieval system** into an **interactive, multi-round recommendation system**.

Instead of returning results once, the system allows users to:
- give feedback (like / dislike / constraints)
- refine preferences
- receive improved recommendations iteratively

---

## 2. Current System (v1 Summary)

Current pipeline:

```
User Query
  → Query Parser
  → TF-IDF Retrieval (cosine similarity)
  → Hard Filters (location, cuisine, distance, price)
  → Multi-factor Re-ranking
  → Top-K Results
```

Scoring function:

```
final_score =
  w_semantic * similarity +
  w_rating * rating_score +
  w_price * price_match +
  w_distance * distance_score +
  w_popularity * popularity_score
```

---

## 3. Key Upgrade Idea

Transform system into:

> Retrieval + Feedback-driven Re-ranking (Interactive Loop)

---

## 4. New System Architecture (v2)

### 4.1 Overall Flow

```
Round 1:
  User Query
    → Retrieval + Ranking
    → Top-K Results

Round N:
  User Feedback
    → Preference Update
    → Candidate Expansion (Top-N pool)
    → Re-ranking with preference
    → New Top-K Results
```

---

## 5. Core Concept: Preference State

Maintain a session-level user preference state:

```python
user_preference = {
  "liked_items": [],
  "disliked_items": [],
  "preferred_cuisines": [],
  "disliked_cuisines": [],
  "price_preference": None,
  "max_distance": None,
  "min_rating": None
}
```

This evolves after each interaction.

## 7. Candidate Expansion Strategy

Instead of only Top-K, keep a larger pool:

```
Initial retrieval → Top 50 candidates
Then:
  → apply re-ranking
  → return Top 5
```

---

## 8. New Scoring Function (v2)

```
new_score =
  a * original_final_score
+ b * similarity_to_liked_items
- c * similarity_to_disliked_items
+ d * preference_match_score
```

---

### 8.1 Similarity to Liked Items

```
sim_liked = average(
  cosine(item_vector, liked_item_vector)
)
```

---

### 8.2 Similarity to Disliked Items

```
sim_disliked = average(
  cosine(item_vector, disliked_item_vector)
)
```

---

### 8.3 Preference Match Score

Includes:

- cuisine match
- price match
- distance constraint
- rating threshold

Example:

```
preference_match =
  cuisine_match * 0.3 +
  price_match * 0.3 +
  distance_score * 0.2 +
  rating_score * 0.2
```

---

## 9. Item-to-Item Recommendation (Important Upgrade)

If user likes a restaurant:

```
Find similar restaurants based on:
  - TF-IDF profile similarity
  - category overlap
  - price tier
  - location proximity
```

Then merge with query-based results.
